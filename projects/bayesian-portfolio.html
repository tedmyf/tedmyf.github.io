Bayesian Portfolio Construction

A heuristic optimizer paired with Bayesian updating, exposed through a slider UI that shows analysts how shifts in prior conviction redistribute weights across asset class and regional views.

Context

Capital market assumptions sit upstream of every multi-asset portfolio. Analysts hold strong priors about asset class returns, volatilities, and correlations. Classic mean-variance optimizers translate those priors into corner solutions and pathological allocations whenever the inputs shift even slightly.

The Black–Litterman framework addresses this by blending an equilibrium prior with subjective views. Standard implementations leave the conviction parameter buried in code, hidden from the analyst making the call. The goal of this project: turn that conviction parameter into a first-class control surface.

Approach

Two components run in tandem. A heuristic optimizer produces a baseline allocation under reference assumptions. A Bayesian update layer revises that allocation toward analyst views, with prior strength controlled by a continuous slider in the UI.

At one extreme the slider returns the heuristic baseline unchanged. At the other, it lets analyst views fully override. Intermediate positions interpolate the posterior, exposing exactly how much each percentage point of conviction is worth in basis points of allocation.

Architecture

The Python service computes both the heuristic baseline and the Bayesian posterior, exposing them as a single JSON payload. A Vue 3 frontend pulls those payloads on slider change, debounced to limit recomputation. Cached posterior outputs live in S3, keyed by view set and prior. The TypeScript types are shared between server and client to keep the schema honest.

Outcomes

Internal use across the asset allocation team. The interactive prior-strength surface reduces the cognitive cost of running scenario analyses and makes the role of the prior parameter visible rather than implicit.

What I Learned

Bayesian methods earn their keep when the conviction parameter is exposed, not hidden. Most production optimizers fail not because the math is wrong but because the inputs are buried three abstractions deep and analysts cannot interrogate them in real time. UI design is part of the model.
