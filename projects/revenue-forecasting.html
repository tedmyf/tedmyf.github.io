Revenue Forecasting

A predictive orders model embedded in the corporate sales and operations planning cycle, built to replace a manual spreadsheet workflow that gated $5B in annual revenue planning.

Context

Sales and operations planning at the corporate level reconciles bookings, capacity, and revenue commitments across divisions. The existing forecast process relied on hand-curated spreadsheets, with cycle times measured in weeks and assumption changes propagated by email. Forecast quality varied with the analyst running the cycle.

The remit: produce a single, auditable predictive orders model that finance, supply chain, and division leadership could agree on, without sacrificing the explanatory clarity that the spreadsheet offered.

Approach

The model regresses incoming orders on a feature set drawn from upstream demand signals, macroeconomic indicators, and seasonal book-to-bill patterns. Variable selection used cross-validated penalized regression to control collinearity across overlapping market signals.

Explainability was a hard requirement. Each forecast point ships with a coefficient-weighted feature contribution table, so a planner can see why the model raised or lowered the next quarter and challenge the assumption in real time.

Implementation

The Python pipeline pulls data from OneStream and MicroStrategy, runs the regression, and writes results to a Power BI semantic model that finance leadership already trusted. No new dashboards. The model lives behind an interface the planners had been using for years; only the inputs changed.

This was a deliberate scope decision. A model that asks finance to switch tools at the same time it asks them to switch methodologies will fail on adoption alone, regardless of accuracy.

Outcomes

The forecasting cycle compressed from a multi-week effort to under a day end-to-end. Out-of-sample accuracy beat the prior process by a meaningful margin on the validation horizon. The model now runs as a recurring step in the corporate planning calendar.

What I Learned

The hardest part of internal forecasting work is not the model. It is the negotiation over which features and assumptions count as legitimate, and the migration of a process that several teams already had reasons to defend. Shipping into the existing tooling, rather than alongside it, was the move that made adoption possible.
